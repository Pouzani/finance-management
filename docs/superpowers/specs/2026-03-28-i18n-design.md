# i18n Design — Finance Management

**Date:** 2026-03-28
**Status:** Approved
**Scope:** Add `next-intl` i18n with URL-based locale routing; migrate all hardcoded strings (French and English) to English translation keys.

---

## 1. Goal

Replace all hardcoded strings across the app with `next-intl` translation keys. The app currently contains a mix of French and English text; everything is standardised to English. The URL structure reflects the active locale (e.g. `/en/dashboard`). Adding future languages (e.g. `fr`, `ar`) requires only a new `messages/<locale>.json` file and a one-line addition to the supported locales list.

---

## 2. Library

**`next-intl`** — purpose-built for Next.js App Router. Provides:
- `useTranslations()` hook that works in both Server and Client Components
- Locale-aware `Link` and `useRouter` (automatically prefixes hrefs with the active locale)
- TypeScript autocompletion for all message keys
- Middleware for locale detection and redirect

---

## 3. Directory Structure

```
app/
  [locale]/
    layout.tsx          ← next-intl provider, sets <html lang={locale}>
    page.tsx            ← redirect to /[locale]/dashboard
    login/page.tsx
    register/page.tsx
    dashboard/
      layout.tsx
      page.tsx
      transactions/page.tsx
      budgets/page.tsx
      goals/page.tsx
      accounts/page.tsx
      reports/page.tsx
      settings/page.tsx
  layout.tsx            ← root layout (fonts, global CSS only, no locale logic)
middleware.ts           ← locale detection + redirect (/ → /en)
i18n/
  request.ts            ← next-intl server-side config (getRequestConfig)
  navigation.ts         ← re-exports locale-aware Link, useRouter, usePathname from createNavigation
messages/
  en.json               ← all English strings
next.config.ts          ← updated with next-intl plugin
```

The root `app/layout.tsx` remains as a thin shell (fonts, global CSS). All locale-aware rendering lives in `app/[locale]/layout.tsx`.

---

## 4. Middleware

`middleware.ts` uses `next-intl/middleware` (or `createNavigation` in next-intl v3+):

- Supported locales: `["en"]` (extend later)
- Default locale: `"en"`
- Behaviour: `/` → `/en`, unknown locales → 404

## 6. Component Usage Pattern

**Server Component:**
```tsx
import { useTranslations } from 'next-intl';

export default function Sidebar() {
  const t = useTranslations('nav');
  return <span>{t('dashboard')}</span>;
}
```

**Client Component** — identical API:
```tsx
'use client';
import { useTranslations } from 'next-intl';

export default function QuickTransaction() {
  const t = useTranslations('transactions');
  // ...
  setError(t('errorLabel'));
}
```

**Locale-aware navigation** — use `next-intl`'s `Link` instead of Next's:
```tsx
import { Link } from '@/i18n/navigation'; // re-exported from next-intl createNavigation
```

## 7. Migration Order

1. **Install & configure** — `next-intl`, `next.config.ts`, `i18n/request.ts`, `middleware.ts`, `messages/en.json`
2. **Restructure routes** — move all files from `app/` into `app/[locale]/`, update root `app/layout.tsx`
3. **Replace strings** — file by file, replace hardcoded strings with `t('key')`, using the namespace that matches the feature
4. **Update navigation** — replace `next/link` imports with locale-aware `Link` from `@/i18n/navigation`; replace `next/navigation` `useRouter` and `usePathname` similarly. Also rename the `Period` type value `"1A"` → `"1Y"` in `lib/months.ts` (French abbreviation → English) and update the `ReportsView` period array accordingly.
5. **Update tests** — tests asserting on French strings (e.g. `"Dépense"`) are updated to English equivalents

---

## 8. What Is NOT in Scope

- Language switcher UI (can be added later)
- RTL layout support for Arabic (structural change, separate task)
- Translating category names stored in the database (those come from the API, not the UI)
- Locale-specific number/date formatting changes (MAD formatting via `formatMAD()` is unchanged)
